# 流程规范违反检讨

**日期**: 2026-03-21  
**问题**: 修复 `python-multipart` 依赖缺失时流程不规范

---

## 1. 问题描述

在修复 `python-multipart` 依赖缺失问题时，主会话直接修改了代码文件，而非委托给编码智能体（@fixer）完成。这违反了主会话"只传达需求不管代码"的原则。

---

## 2. 违反的流程规范

根据 AGENTS.md 中的工作流程定义：

> **主会话职责**: 理解需求 → 分析路径 → 委托执行 → 验证结果

主会话应当：
- 解析用户请求，明确需求
- 评估最佳执行路径
- 委托给合适的专家智能体执行
- 验证结果是否符合要求

**违规行为**：主会话直接执行了代码修改，而非委托给 @fixer 智能体。

---

## 3. 正确流程应该是什么

### 正确流程

```
用户请求 → 主会话理解需求 → 分析任务类型 → 委托 @fixer 执行 → 验证结果
```

### 具体步骤

1. **理解需求**: 识别到这是一个代码修复任务
2. **路径分析**: 
   - 任务明确（添加缺失依赖）
   - 有清晰的修复方案
   - 适合委托给 @fixer
3. **委托执行**: 应调用 @fixer 智能体，传递完整上下文：
   - 问题描述：`python-multipart` 依赖缺失
   - 修复方案：在 `pyproject.toml` 中添加依赖
   - 验证命令：运行相关测试确认修复
4. **验证结果**: 检查 @fixer 的输出，确认修复成功

---

## 4. 启示和改进措施

### 启示

1. **角色边界清晰**: 主会话是"指挥官"，不是"执行者"。混淆角色会导致流程混乱。
2. **委托效率**: 对于明确的代码修改任务，@fixer 可以并行处理多个独立任务，效率更高。
3. **可追溯性**: 通过智能体委托，所有操作都有清晰的记录和上下文。

### 改进措施

1. **强制委托检查**: 在执行任何代码修改前，先问自己"这个任务应该由谁执行？"
2. **任务分类**:
   - 研究类任务 → @explorer / @librarian
   - 架构决策 → @oracle
   - UI/UX 设计 → @designer
   - 代码执行 → @fixer
3. **并行化意识**: 当有 3+ 个独立任务时，应考虑并行委托多个 @fixer 实例。

---

## 总结

本次违规的根本原因是主会话越俎代庖，直接执行了本应由 @fixer 完成的工作。今后必须严格遵守"理解 → 分析 → 委托 → 验证"的工作流程，确保每个智能体各司其职。
